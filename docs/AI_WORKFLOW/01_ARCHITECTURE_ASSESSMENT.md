# 01_ARCHITECTURE_ASSESSMENT: تقييم البنية المعمارية الحالية

تقدم هذه الوثيقة تقييماً تفصيلياً ومعمارياً كاملاً لهيكل ونظام **Khidmah Pro** (برمجياً **Carag V2**)، لتوضيح نقاط القوة والضعف والقيود المعمارية للمطورين والمهندسين.

---

## 1. الملخص المعماري (Architectural Summary)

يعتمد نظام Khidmah Pro على معمارية هجينة متطورة تجمع بين كفاءة التطوير في **Laravel 12** وديناميكية وحيوية واجهات **Vue 3**، باستخدام **Inertia.js 2.x** كجسر نقل للبيانات بدون بناء REST API مستقل لإدارة واجهات الويب.

```
┌────────────────────────────────────────────────────────┐
│                      Vue 3 SPA                         │
│   (Pages / Components / Composables / Layouts / i18n)  │
└──────────────────────────┬─────────────────────────────┘
                           │ (Inertia SPA Bridge)
┌──────────────────────────▼─────────────────────────────┐
│                   Laravel Controllers                  │
│       (App Controllers / System / Public / API)        │
└──────────────────────────┬─────────────────────────────┘
                           │ (Actions / Services)
┌──────────────────────────▼─────────────────────────────┐
│                 Eloquent Models & Scopes               │
│         (TenantScoped / CenterScoped / Policies)        │
└──────────────────────────┬─────────────────────────────┘
                           │
┌──────────────────────────▼─────────────────────────────┐
│                     MySQL 8 Database                   │
└────────────────────────────────────────────────────────┘
```

## 2. تفصيل الطبقات البرمجية (Layer Analysis)

### أ. طبقة النماذج والبيانات (Model & Data Layer)
* **عزل المستأجرين**: يتم العزل منطقياً في نفس قاعدة البيانات عبر سمات Eloquent العالمية:
  * **`TenantScoped`**: يفرض تلقائياً تصفية الاستعلامات بـ `tenant_id` للمستأجر الحالي.
  * **`CenterScoped`**: يفرض تصفية إضافية بـ `center_id` للمستخدم الحالي لعزل الفروع مع السماح بجلب السجلات العامة للنظام التي تحمل حقل `source = 'system'`.
* **الضمان والاستقرار المالي**: يتم استخدام سمة `HasTaxSnapshot` لحفظ لقطة ضريبية ثابتة تاريخياً تمنع تغير قيم الفواتير الصادرة عند تعديل إعدادات المنشأة لاحقاً.

### ب. طبقة منطق الأعمال (Services & Actions)
* يتبع النظام **Service/Action Pattern** بدلاً من Repository Pattern:
  * **الخدمات (Services)**: مثل `InventoryService` لإدارة العمليات المستودعية و `InvoiceService` لتحويل كروت العمل لفواتير وحساب الضرائب وتوليد نصوص الـ QR لـ ZATCA بصيغة TLV.
  * **العمليات المنفصلة (Actions)**: فئات تؤدي وظيفة واحدة معقدة مثل `MergeCustomerAction` لدمج العملاء وتصحيح التكرار و `CreateWorkOrderAction` لإنشاء كروت العمل.

### ج. طبقة العرض والتحكم (Controllers & Routing)
* **المتحكمات (Controllers)**: مقسمة بشكل نظيف في مجلدات منفصلة بناءً على نوع المستخدم والصلاحيات:
  * `App/`: لإدارة العمليات اليومية للمراكز (Customers, Vehicles, WorkOrders, Invoices, Inventory).
  * `System/`: لمدراء النظام (Super Admins) للتحكم بالاشتراكات والمنصة.
  * `Public/`: للصفحات العامة غير الموثقة (عرض مقايسات الأسعار للعملاء عبر UUID).
  * `Api/`: لاستقبال بيانات أجهزة البصمة.
  * `Auth/`: لإدارة الدخول والتحقق والـ 2FA.

## 3. تقييم الديون التقنية والقيود (Technical Debt & Limitations)

من خلال مراجعة الكود الحالي، تم رصد التحديات المعمارية التالية:

### 1. المتحكمات الضخمة (Fat Controllers)
* **المشكلة**: ملف `WorkOrderController.php` حجمه يقارب 80 كيلوبايت ويحتوي على أكثر من 1200 سطر كود. يجمع بين إدارة كروت العمل، تحديث الحالات، صرف قطع الغيار، ورفع الصور، مما يمثل عبئاً كبيراً في الصيانة ويخالف مبدأ المسؤولية الفردية (SRP).
* **الأثر**: صعوبة إجراء اختبارات آلية تفصيلية للمتحكم وزيادة احتمالية حدوث أخطاء (Bugs) عند تعديل أي جزء منه.

### 2. تشتت منافذ الـ API الداخلية (Route Clutter)
* **المشكلة**: مسارات البحث والإكمال التلقائي للقطع والعملاء مكتوبة داخل ملف مسارات الويب الرئيسي `web.php` وتوجه إلى App Controllers بدلاً من عزلها في ملف مسارات مستقل أو استخدام متحكمات API مخصصة.
* **الأثر**: زيادة حجم ملف `web.php` وتشتت الفحص الأمني لمعدل الطلبات (Rate Limiting).

### 3. غياب طبقة التجريد لقواعد البيانات (No Repository Layer)
* **المشكلة**: يتم كتابة استعلامات Eloquent معقدة وعمليات الربط مباشرة داخل الـ Controllers والـ Services دون وجود طبقة وسيطة للتجريد (Repositories).
* **الأثر**: يصعب هذا من عزل منطق الأعمال لعمل اختبارات وحدة (Unit Tests) نظيفة دون الاتصال الحقيقي بقاعدة البيانات، ويصعب الانتقال لمحرك قاعدة بيانات آخر مستقبلاً إذا دعت الحاجة.

### 4. قصور في هيكلة أجهزة البصمة والجدولة
* **المشكلة**: لا توجد طبقة معالجة مهام خلفية مستقلة (Jobs/Queues) تعتمد على Redis للتعامل مع الإدخال الكثيف لبيانات البصمة، وتعتمد حالياً على قاعدة البيانات مباشرة مما قد يسبب بطء النظام في الورش الضخمة التي تحتوي على مئات الموظفين.

---

## 4. التوصيات والتوجهات المستقبلية (Architectural Recommendations)

1. **تفكيك `WorkOrderController`**: يجب نقل العمليات المعقدة إلى فئات Actions مخصصة وإبقاء الـ Controller كطبقة توجيه واستدعاء فقط.
2. **عزل مسارات الـ API**: نقل كافة مسارات الإكمال التلقائي والبحث إلى `routes/api.php` أو حمايتها بمجموعات برمجيات وسيطة (Middleware Groups) مخصصة للتحقق وتحديد معدل الاستدعاء.
3. **فصل معالجة الحضور الكثيف**: تحويل الطوابير (Queues) إلى Redis في البيئات الإنتاجية لمعالجة بيانات البصمة الواردة بشكل متزامن دون التأثير على أداء لوحة تحكم المركز.
