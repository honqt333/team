# 06_API_RULES: قواعد ومعايير تصميم الواجهات البرمجية (APIs)

تحدد هذه الوثيقة المعايير الفنية لتصميم وتطوير واجهات برمجة التطبيقات (APIs) الخارجية والداخلية لنظام **Khidmah Pro**.

---

## 1. هيكلة المسارات والترميز (Routing & Standards)

يتم تقسيم الـ APIs في المشروع إلى نوعين رئيسيين بناءً على الاستخدام:

### أ. الواجهات البرمجية الخارجية (External/Public APIs)
* **المسار**: يجب كتابتها داخل ملف `routes/api.php` وتغليفها بإصدار واضح تحت البادئة `/api/v1/`.
* **مثال**: مسارات أجهزة البصمة:
  ```php
  Route::prefix('v1/attendance')->group(function () {
      Route::post('/clock', [BiometricAttendanceController::class, 'clock']);
  });
  ```
* **الحماية**: تُحمى هذه الواجهات باستخدام حارس مصادقة التوكنات (API Token Bearer Authentication) المخصصة للأجهزة والأنظمة الخارجية.

### ب. الواجهات البرمجية الداخلية المساعدة (Internal Autocomplete/Data APIs)
* **المسار**: تُستخدم لتغذية مكونات Vue بالبيانات (الإكمال التلقائي للعملاء، البحث عن القطع، فحص الرصيد).
* **التوجيه**: تُكتب داخل `routes/web.php` تحت حارس الجلسة الموحد للمستخدم للتأكد من مشاركة سياق المستأجر والفرع تلقائياً.
* **البادئة**: يفضل تسميتها بـ `/api/` أو `/app/api/` لتمييزها عن مسارات صفحات Inertia.

---

## 2. معايير الاستجابة والأخطاء (JSON Response & Error Standards)

يجب أن ترجع كافة الـ APIs استجابة موحدة بصيغة JSON مع كود الحالة HTTP المناسب:

### أ. استجابة العمليات الناجحة (Success Response)
تتضمن حقل الحالة الأساسي والبيانات المستهدفة:
```json
{
    "success": true,
    "data": {
        "id": 123,
        "name": "Customer Name"
    }
}
```

### ب. استجابة العمليات الفاشلة والأخطاء (Error Response)
تتضمن رمز الخطأ المعياري ورسالة واضحة للمستخدم:
```json
{
    "success": false,
    "error": "employee_not_found",
    "message": "Employee not found with the provided identifier"
}
```

## 3. أكواد الحالة HTTP المعتمدة (HTTP Status Codes)

يجب الالتزام بالاستخدام الدقيق لأكواد الحالة التالية:

| كود الحالة | الاسم المعياري | الاستخدام في النظام |
| :--- | :--- | :--- |
| **`200`** | OK | نجاح عمليات جلب البيانات أو التحديث. |
| **`201`** | Created | نجاح إنشاء سجل جديد بنجاح. |
| **`400`** | Bad Request | خطأ في إرسال المعاملات أو طلب غير منطقي. |
| **`401`** | Unauthenticated | طلب غير موثق (توكن مفقود أو غير صالح). |
| **`403`** | Forbidden | المستخدم موثق ولكنه لا يملك الصلاحية للعملية. |
| **`404`** | Not Found | المورد المطلوب غير موجود أو محذوف. |
| **`429`** | Too Many Requests | تجاوز الحد المسموح به لمعدل الاستدعاء (Rate Limit). |
| **`422`** | Unprocessable Entity | فشل التحقق من الحقول ونماذج الإدخال (Validation Fail). |

---

## 4. تدابير الحماية ومعدل الاستدعاء (Rate Limiting & Security)

* **تحديد معدل الطلبات**: يجب حماية مسارات الـ API الحساسة بـ Middleware يحدد معدل الطلبات لتقليل محاولات إغراق خوادم المنصة.
  * مثال: تحديد 30 طلباً في الدقيقة لأجهزة البصمة: `middleware('throttle:30,1')`.
* **الالتزام بالصلاحيات**: يجب استدعاء السياسات الأمنية `$this->authorize('viewAny', Model::class)` داخل دوال الـ API للتأكد من أن التوكن المستخدم لديه حق الوصول للمورد.
