# 04_EXTENSION_LAYER: معايير طبقة التمديد والإضافات البرمجية

تحدد هذه الوثيقة آليات إضافة وتفعيل الميزات الإضافية (Modules / Add-ons) وتمديد منطق العمل والواجهات في نظام **Khidmah Pro** بطريقة تضمن بقاء نواة النظام (Core) نقية ودون تداخل يضر بالتحديثات البرمجية المستقبلية للأساس.

---

## 1. تمديد قواعد البيانات والنماذج (Eloquent Models Extension)

عند إضافة حقول أو علاقات لنماذج البيانات المستقرة (مثل `Customer`, `Vehicle`, `WorkOrder`)، يمنع تعديل ملفات النماذج الأساسية مباشرة. اتبع الأساليب التالية:

### أ. الجداول الملحقة والعلاقات (Joined Tables)
* بدلاً من إضافة حقول جديدة مباشرة للجدول الرئيسي، قم بإنشاء جدول ملحق باسم يعكس الوظيفة الإضافية (مثال: `work_order_inspections` أو `vehicle_mileage_logs`).
* اربط الجدول الملحق بعلاقة Eloquent مناسبة (`hasOne`, `hasMany`) مع النموذج الأساسي.

### ب. النطاقات المخصصة (Query Scopes)
* عند كتابة تصفية مخصصة للبيانات، استخدم الـ Local Scopes داخل النموذج الأساسي لتسهيل استدعائها.
* مثال: تصفية كروت العمل الجاهزة للفحص الفني:
  ```php
  public function scopeReadyForQc($query)
  {
      return $query->where('status', 'ready_for_qc');
  }
  ```

## 2. تمديد منطق الأعمال (Business Logic Extension)

يعتمد تمديد العمليات المالية والتشغيلية على عزل المسؤوليات باستخدام العمليات الفردية (**Actions**) والخدمات المغلفة (**Decorators**):

```
┌────────────────────────────────────────────────────────┐
│                   Controller Request                   │
└──────────────────────────┬─────────────────────────────┘
                           │
┌──────────────────────────▼─────────────────────────────┐
│                 Decorator / Action Class               │
│         (Wraps & extends core business rules)          │
└──────────────────────────┬─────────────────────────────┘
                           │
┌──────────────────────────▼─────────────────────────────┐
│                      Core Service                      │
│        (Executes stable transactions: e.g. WAC)       │
└────────────────────────────────────────────────────────┘
```

### أ. فئات العمليات المنفصلة (Action Classes)
* تُنشأ الفئة داخل مجلد `app/Actions/[Domain]/` وتكون مسؤولة عن تنفيذ عملية واحدة فقط (مثل: `MergeCustomerAction`).
* يتم تمرير البيانات اللازمة للفئة واستدعاؤها داخل الـ Controller، مما يقلل من حجم أكواد الـ Controllers ويسهل عمل اختبارات الوحدة لها.

### ب. تغليف وتزيين الخدمات (Decorator Pattern)
* إذا كان لديك منطقاً إضافياً يجب تنفيذه بالتزامن مع خدمة مستقرة (مثل صرف قطع غيار من المخزن وتحديث تقرير خارجي)، قم بإنشاء فئة تغلف الخدمة الأساسية وتضيف السلوك الجديد قبل أو بعد الاستدعاء الأساسي، دون تعديل كود الخدمة الأساسية `InventoryService`.

## 3. تمديد الواجهات الأمامية (Frontend Extension Layer)

لتجنب حدوث تضخم في ملفات صفحات الواجهة الرئيسية (مثل صفحة تفاصيل كرت العمل `WorkOrders/Show.vue`) عند تفعيل ميزات جديدة:

1. **استخلاص المكونات (Component Extraction)**: يجب تقسيم الصفحة إلى مكونات فرعية صغيرة قابلة للتركيب متواجدة داخل مجلد فرعي باسم الموديول (مثل: `Components/WorkOrders/`).
2. **عزل الحالات في Composables**: يتم الاحتفاظ بالبيانات التفاعلية (Reactive State) وعمليات الاستدعاء للـ APIs في ملفات Composable خارجية (مثل `useWorkOrderItems.js`).
3. **تفعيل الميزات عبر Feature Flags**: يتم فحص أحقية المستأجر في استخدام الميزة البرمجية المتقدمة (مثل التقسيط تابي/تمارا) قبل رندرة المكون بالواجهة الأمامية باستخدام دالة التحقق:
  ```js
  v-if="$page.props.tenant.modules.payments_bnpl"
  ```
