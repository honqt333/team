# 03_GAP_ANALYSIS: تحليل الثغرات والميزات الناقصة للنظام

توضح هذه الوثيقة الفجوة (Gap) بين الحالة الحالية للنظام والخصائص البرمجية المطلوبة للوصول بـ **Khidmah Pro** إلى نظام تخطيط موارد مؤسسات (**ERP**) كامل متكامل وقابل للتسويق على نطاق واسع.

---

## 1. الفجوة الأولى: بوابة الفنيين المتنقلة (Technician Portal - `/app/my`)

### الوضع الحالي
توجد هيكلية برمجية أساسية لبوابة الموظف في مجلد `EmployeePortal` تعرض الحضور والإجازات والرواتب، لكنها تفتقر لواجهة فنية تشغيلية مخصصة للتقنيين داخل الورشة لإدارة مهام الصيانة اليومية.

### المتطلبات المفقودة لإكمال الفجوة
* **لوحة المهام الفنية (My Tasks)**: عرض المهام والخدمات المسندة للفني الحالي داخل كرت العمل فقط دون إظهار أي تفاصيل مالية أو أسعار (امتثالاً لسرية معلومات المؤسسة).
* **تحديث الحالات اللحظي**: إمكانية بدء العمل (Start) أو تعليق المهمة (Suspend) أو إكمالها (Complete) مباشرة من الجوال/التابلت لتحديث حالة بند أمر العمل آلياً.
* **الفحص ورفع المرفقات**: تمكين الفني من التقاط صور للأجزاء التالفة بالسيارة ورفعها مباشرة من هاتفه لترتبط بتبويب الصور الخاص بأمر العمل، وكتابة ملاحظات الفحص الفني.

---

## 2. الفجوة الثانية: نظام الحجوزات المتقدم والجدولة (Advanced Booking & Calendar)

### الوضع الحالي
لا يوجد محرك جدولة تفاعلي؛ يتم تسجيل معلومات المركبات عند دخولها للورشة بشكل فوري ويدوي في أمر العمل.

### المتطلبات المفقودة لإكمال الفجوة
* **مخطط المواعيد (Calendar View)**: واجهة تقويم تفاعلية تعرض المواعيد المفتوحة والمحجوزة وتوزيعها على الفروع ورافعات الفحص المتوفرة.
* **بوابة الحجز العامة للعميل**: صفحة ويب عامة تابعة لكل مستأجر تسمح لعملائه باختيار الفرع، وتحديد الخدمة المطلوبة، واختيار التاريخ المتاح لحجز موعد.
* **التحويل الذكي**: تحويل الحجز المعتمد إلى أمر عمل (Work Order) بضغطة زر واحدة مع سحب كافة بيانات العميل والسيارة والخدمات المحددة مسبقاً.

---

## 3. الفجوة الثالثة: الربط الإلكتروني للفوترة - المرحلة الثانية (ZATCA Phase 2)

### الوضع الحالي
يدعم النظام متطلبات المرحلة الأولى فقط من الفوترة الإلكترونية السعودية (توليد QR Code بصيغة TLV وتضمينه في الفاتورة المبسطة محلياً).

### المتطلبات المفقودة لإكمال الفجوة
* **توليد فاتورة XML**: تحويل بيانات الفاتورة إلى ملف XML متوافق مع معايير الهيئة (UBL 2.1).
* **التوقيع الرقمي والتشفير (Cryptographic Signing)**: تشفير الفاتورة باستخدام الختم الرقمي الخاص بالمنشأة وتوليد الهاش وتمريره.
* **التسلسل والترابط (Chaining)**: ربط الفاتورة الحالية بهاش الفاتورة السابقة لمنع التلاعب وتوليد قيمة ICV المتزايدة.
* **الإرسال والتسجيل (Clearance / Reporting)**: الربط المباشر بـ API منصة (Fatoora) الحكومية لإرسال الفواتير وتسجيلها لحظياً.

---

## 4. الفجوة الرابعة: النظام المحاسبي المتكامل (Accounting General Ledger)

### الوضع الحالي
يقتصر النظام المالي حالياً على جداول المبيعات (الفواتير الصادرة)، والمشتريات (الفواتير الواردة)، وجدول للمعاملات المباشرة المبسطة `company_transactions` مع تصنيفاتها `income_categories`.

### المتطلبات المفقودة لإكمال الفجوة
* **شجرة الحسابات (Chart of Accounts)**: إعداد نظام محاسبي كامل يحتوي على الأصول، الخصوم، حقوق الملكية، الإيرادات، والمصروفات.
* **القيد المزدوج (Double-Entry Bookkeeping)**: توليد قيود اليومية آلياً عند إصدار فاتورة أو سداد دفعة أو صرف قطع غيار لتسجيلها في الحساب المالي المقابل.
* **التقارير المالية المتقدمة**: توليد ميزان المراجعة، قائمة الدخل، والميزانية العمومية تلقائياً للمنشأة وفروعها.

---

## 5. الفجوة الخامسة: التواصل التلقائي مع العملاء (CRM Alerts & Notifications)

### الوضع الحالي
يوجد موديل لإعداد قوالب التواصل وصلاحيات إرسال الإشعارات، لكن لا توجد أتمتة لربط الأحداث بالرسائل الخارجية للعميل.

### المتطلبات المفقودة لإكمال الفجوة
* **أتمتة الرسائل**: الربط مع WhatsApp Business API أو مزودي الـ SMS لإرسال رابط عرض السعر للعميل تلقائياً فور إعداده، وإرسال رابط الفاتورة والدفع الإلكتروني فور الانتهاء من العمل.
* **تذكيرات المتابعة**: إرسال تذكيرات تلقائية للعملاء الذين لديهم فواتير معلقة السداد، أو إرسال عروض ترويجية للعملاء الذين لم يزوروا المركز منذ فترة طويلة.
