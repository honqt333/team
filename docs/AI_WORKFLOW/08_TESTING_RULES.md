# 08_TESTING_RULES: قواعد وكتابة الاختبارات البرمجية الآلية

تحدد هذه الوثيقة المعايير والبروتوكولات الإلزامية لكتابة وتنفيذ الاختبارات البرمجية (Testing Suite) لضمان عدم حدوث تراجعات (Regressions) وحماية أمان النظام والبيانات.

---

## 1. هيكل الاختبارات وأمر التشغيل (Testing Structure)

يعتمد النظام على **PHPUnit** كإطار عمل رئيسي للاختبارات، ويتم تقسيم الاختبارات إلى نوعين أساسيين:

* **الاختبارات الوظيفية (Feature Tests)**: تُحفظ في المجلد `tests/Feature/` وتُستخدم لاختبار صلاحيات مسارات التحكم (Controller Routes)، وسياسات الأمان (Policies)، وعزل البيانات للمستأجرين، وتكامل العمليات المالية.
* **اختبارات الوحدة (Unit Tests)**: تُحفظ في المجلد `tests/Unit/` وتُستخدم لاختبار العمليات الرياضية الحسابية (مثل حساب الضرائب والنسب المئوية) أو منطق عمل فئة منفردة (مثل `CreateWorkOrderAction`) بشكل مستقل تماماً عن قاعدة البيانات أو الخادم.

### أمر تشغيل الاختبارات الرئيسي
لتشغيل حزمة الاختبارات بالكامل قبل رفع التعديلات:
```bash
php artisan test
```

---

## 2. قواعد اختبار عزل المستأجرين (Tenancy Isolation Testing)

يجب كتابة اختبار عزل المستأجرين لكل ميزة جديدة تتعامل مع بيانات المستأجرين لضمان عدم إمكانية تسريب البيانات:

1. قم بإنشاء مستأجرين وهميين (Tenant A و Tenant B) مع الفروع المقابلة لهما.
2. قم بإنشاء سجل مستهدف (مثل عميل أو فاتورة أو أمر عمل) ينتمي للمستأجر A.
3. قم بتوثيق الدخول (Authenticate) كمستخدم تابع للمستأجر B.
4. حاول إرسال طلب عرض أو تعديل أو حذف للسجل التابع للمستأجر A.
5. **النتيجة المتوقعة**: يجب تأكيد رفض الطلب ووصول استجابة بالمنع كود `403 Forbidden` أو عدم العثور على المورد `404 Not Found`.

---

## 3. معايير محاكاة الخدمات الخارجية (Mocking Rules)

يُمنع منعاً باتاً استدعاء خوادم إنتاجية أو تجريبية حقيقية للخدمات الخارجية أثناء تشغيل الاختبارات الآلية. يجب استخدام محاكاة برمجية (**Mocking**):

* **بوابات الدفع (Tabby, Tamara, Moyasar)**: محاكاة فئات الاستجابة ورموز الخطأ باستخدام `Http::fake()` أو عبر حقن واجهات وهمية لمدير الدفع `PaymentManager`.
* **خدمات المراسلة (SMS & WhatsApp API)**: محاكاة خدمات الإرسال باستخدام نظام التزييف الافتراضي لـ Laravel (`Notification::fake()` أو `Mail::fake()`).
* **أجهزة البصمة**: محاكاة استدعاء منفذ البصمة بإرسال طلب POST وهمي برأس توثيق مخصص والتحقق من النتيجة في قاعدة بيانات الاختبار.

---

## 4. إعدادات قاعدة بيانات الاختبارات (Test Database Configuration)

* يتم تشغيل الاختبارات باستخدام قاعدة بيانات منفصلة (SQLite أو MySQL مخصصة للاختبارات) تم تهيئتها في ملف `phpunit.xml`.
* يجب استخدام السمة `RefreshDatabase` أو `DatabaseTransactions` لإعادة تهيئة قاعدة البيانات والتراجع عن التغييرات بعد كل فحص لضمان نظافة الاختبارات اللاحقة.
