# ADR-0002: Multi-Tenancy Strategy - Logical Database Isolation

## 1. الحالة (Status)
**مقبول (Accepted)**

---

## 2. السياق (Context)
يتطلب تشغيل النظام كمنصة سحابية متعددة المستأجرين (SaaS) عزل بيانات كل منشأة (Tenant) بشكل كامل وصارم، مع دعم إمكانية إدارة المنشأة لفروع متعددة (Multi-Branch/Centers) والتنقل بينها.

الخيارات البديلة كانت:
1. **قاعدة بيانات مستقلة لكل مستأجر (Database-per-Tenant)**: إنشاء قاعدة بيانات منفصلة لكل شركة تسجل بالنظام. (يوفر أعلى درجات العزل والأمان، ولكنه معقد جداً في الصيانة والتحديثات الدورية للجداول ويزيد من تكلفة البنية التحتية).
2. **العزل المنطقي بجدول موحد (Single Database Multi-Tenancy)**: عزل كافة المنشآت منطقياً داخل قاعدة بيانات واحدة مشتركة عبر تصفية الاستعلامات بمعرف المستأجر `tenant_id`.

---

## 3. القرار (Decision)
تم اعتماد استراتيجية **العزل المنطقي الموحد (Single Database Logical Multi-Tenancy)** للأسباب التالية:
* سرعة تنفيذ التحديثات البرمجية وهجرات قاعدة البيانات (Database Migrations) دفعة واحدة لكل العملاء.
* تقليل تكلفة تشغيل وإدارة البنية التحتية للخوادم.
* تمكين العزل التلقائي والشفاف عبر طبقة Eloquent Models باستخدام:
  * السمة **`TenantScoped`**: تضيف تلقائياً شرط تصفية عالمي `tenant_id` على كل عملية استعلام وتملأ العمود آلياً عند الحفظ.
  * السمة **`CenterScoped`**: تضيف عزل الفرع `center_id` للموظفين لمنع تداخل العمليات التشغيلية بين الفروع المختلفة لنفس الشركة.
  * التحقق عبر الميدل وير **`EnsureCenterContext`** لفرض تحديد سياق الفرع الفعال `current_center_id` في جلسة المستخدم.

---

## 4. التبعات (Consequences)
* **الإيجابيات**:
  * سهولة التوسع (Scalability) واستضافة آلاف المستأجرين الجدد على نفس قاعدة البيانات فوراً.
  * سهولة التحديثات الهيكلية لقاعدة البيانات وتوحيدها.
  * كفاءة عالية في إدارة الموارد وصيانة خادم البيانات.
* **السلبيات / المخاطر**:
  * خطر تسريب البيانات إذا تم كتابة استعلامات يدوية (Raw SQL) أو استدعاءات تتجاوز النطاقات الافتراضية لـ Eloquent.
  * **الحل**: تم فرض بروتوكول فحص صارم ومراجعة دورية لضمان عدم تجاوز Scopes المحددة وكتابة اختبارات آلية للعزل (`TenancyIsolationTest`).
