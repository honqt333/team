# ADR-0001: Core Architecture - Laravel, Vue 3, and Inertia.js

## 1. الحالة (Status)
**مقبول (Accepted)**

---

## 2. السياق (Context)
يحتاج نظام **Khidmah Pro** إلى بنية برمجية تضمن سرعة التطوير، وكفاءة الأداء، وتجربة مستخدم تفاعلية وسلسة تشبه تطبيقات الصفحة الواحدة (SPA)، مع المحافظة على الحماية الصارمة والتعامل السهل مع قاعدة البيانات في الـ Backend. 

الخيارات البديلة كانت:
1. **فصل كامل (Decoupled SPA & API)**: بناء واجهة أمامية مستقلة (مثل Nuxt.js) تتصل بـ Laravel REST API عبر Sanctum. (يتطلب جهداً مضاعفاً في بناء واجهات الـ API والمصادقة وإدارة الجلسات).
2. **صفحات Laravel التقليدية (Blade Templates)**: استخدام قوالب Blade مع jQuery أو Alpine.js. (يفتقر للتفاعلية والسرعة المطلوبة لإدارة جداول كروت الصيانة والرواتب الضخمة).

---

## 3. القرار (Decision)
تم اعتماد البنية المعمارية الموحدة المعتمدة على:
* **Laravel 12.x** كإطار عمل للخلفية (Backend) لإدارة قواعد البيانات، الصلاحيات، الحماية، والعمليات الحسابية.
* **Vue 3 (Composition API)** حصراً كإطار عمل للواجهات الأمامية، ومنع استخدام الـ Options API لضمان كفاءة تنظيم الأكواد وإعادة الاستخدام عبر الـ Composables.
* **Inertia.js 2.x** كجسر ربط (SPA Bridge) لإرسال واستقبال البيانات والتحكم بالصفحات مباشرة من الـ Controllers، مما يلغي الحاجة لبناء طبقة API معقدة أو إدارة الحالات بشكل منفصل بالواجهة.
* **Tailwind CSS 3/4** كمعيار وحيد للتنسيق البصري ودعم الوضع الداكن (Dark Mode).

---

## 4. التبعات (Consequences)
* **الإيجابيات**:
  * تسريع عملية التطوير بنسبة تزيد عن 40% لعدم وجود حاجة لتكرار كتابة الـ Routes أو التحقق (Validation) بالجهتين.
  * أمان عالي لكون جلسات المستخدم وحمايتها تدار مباشرة بالخلفية عبر Laravel Session.
  * واجهة تفاعلية ممتازة للعميل دون الحاجة لإعادة تحميل الصفحات.
* **السلبيات / القيود**:
  * الارتباط الوثيق بين الواجهة والخلفية؛ لا يمكن استبدال واجهة الويب بتطبيق جوال معتمد على نفس مسارات Inertia مباشرة، مما سيتطلب بناء مسارات API مستقلة لاحقاً للنسخ المحمولة الحقيقية.
